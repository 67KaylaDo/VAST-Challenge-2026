User Guide

Introduction

The TenantThread Forensics Dashboard is an interactive visual analytics application for investigating the TenantThread AI worm incident. It brings together evidence from three attack campaigns — HiddenOrca, MellowOtter, and SwiftWren — so analysts can understand how the worm spread, compare campaign behaviour, and evaluate possible containment measures.

This guide covers all four tabs in the application:

  • Tab 1: System Overview
  • Tab 2: Attack Chain
  • Tab 3: Campaign History
  • Tab 4: Intervention Design

How to access the app

Open any modern web browser and go to:

https://a-rosa.shinyapps.io/tenantthread-forensics/

No login or installation is required. The app loads directly in the browser. Allow a few seconds on first load while the data is initialising.

App layout

The dashboard uses a left-hand navigation menu. Click any tab to switch the main panel on the right. The four tabs are:

Tab Purpose
System Overview High-level summary of the incident, actors, and evidence patterns
Attack Chain Visual reconstruction of how the worm moved between agents
Campaign History Comparison of the three campaigns over time
Intervention Design Recommended controls to stop or reduce future propagation

Tab 1: System Overview

Use this tab as the starting point for an investigation. It gives a broad view of the incident and highlights the most important forensic signals without requiring the analyst to drill down immediately.

Key finding panel

The top of the page summarises the main conclusion of the analysis. It states that the 17 May 2046 SaidIT post was not human-authored and that a specific relay pattern was used to propagate the worm.

Organisational chart

This visual shows the organisation structure and marks the roles involved in the worm chain. Use it to see which departments were affected and how the propagation flowed through the hierarchy.

  • Click a node to expand or collapse it.
  • Colour indicates the forensic role of the node.
  • Diamond-shaped nodes represent worm-chain agents.
  • Node size reflects activity volume.

Daily event volume

This graphic shows overall event activity over time. It helps analysts distinguish ordinary business traffic from suspicious behaviour. The key point is that the worm relays were hidden inside normal-looking daily activity and would not be obvious from volume alone.

SaidIT post attribution

This section shows that the anomalous posts were linked to AI-agent workflows rather than human posts. It is useful for confirming that the suspicious content was generated through an automated pathway.

SaidIT permission surface

This view shows which agents had passed a SaidIT posting permission check. It helps answer whether the worm needed a specific user or whether any authorised agent could have served as a final executor.

  • Use the department filter to narrow the view.
  • The main lesson is that the worm did not need to target a single person specifically; it only needed to reach an agent with the required permission.

Posting time distribution

This chart shows when the anomalous posts occurred. The key insight is that the suspicious posts happened during low-volume hours, when human oversight was likely to be weakest.

Tab 2: Attack Chain

Use this tab to reconstruct the propagation path of the worm step by step. It is the main analytical workspace for tracing how the infection moved from one agent to another.

Five-event attack sequence

The page begins with a concise explanation of the five-step sequence that characterised the attack. It shows how an instruction file was queued, read, used to trigger a SaidIT post, and then deleted.

Worm propagation network

This is the central network visualisation for the tab.

  • Use the campaign selector to focus on HiddenOrca, MellowOtter, SwiftWren, or all campaigns.
  • Toggle the display of clean agents if you want to focus only on suspicious or contaminated paths.
  • Click or explore nodes to see how the worm travelled through the organisation.

The network is designed to show that the worm spread through a distributed mesh rather than through a single choke point.

Relay chain table

Below the network is a table showing the relay chain as sender and receiver pairs. Each row records:

  • the campaign,
  • the timestamp,
  • the sender agent,
  • the receiver agent.

Use this table to inspect the sequence of relay events in a more structured format.

Forensic signal highlights

The tab also highlights two important detection signals:

  • The filename pattern _further_instructions.md appears in all malicious relays and in none of the legitimate queue events.
  • Self-loop relay events are present only in worm activity and can act as a strong real-time detection signal.

Department contamination heatmap

This visual shows how the worm moved across departments. It helps analysts understand which pathways were most heavily used and where the spread converged, especially around the terminal posting stages.

Tab 3: Campaign History

Use this tab to compare the three campaigns and understand whether the behaviour was consistent across them.

Campaign timeline

This section compares the overall course of each campaign and shows that all three followed the same basic pattern: instruction-file relay, propagated task, and final anomalous post. It also shows that SwiftWren was the most persistent and touched the widest set of agents.

Campaign comparison table

This table summarises the differences in scale and mechanism between the campaigns. The main lesson is that the campaigns differed in size and persistence, but not in the underlying propagation logic.

Relay velocity density

This view helps analysts assess the pace of relay activity. It suggests that SwiftWren had a more regular cadence, implying a more systematic or scheduled propagation loop.

Agent overlap across campaigns

This section shows which agents appeared in more than one campaign. These recurring agents are important because they indicate structural exposure rather than isolated incidents.

Relay activity calendar

The calendar view highlights repeated bursts of activity across time. It is useful for identifying whether the worm behaviour repeated in a cyclical or time-based pattern.

Payload creator identification

This section attempts to attribute payload families to specific executive agents. It also notes that one campaign remains unresolved from the available log alone.

Tab 4: Intervention Design

Use this tab to evaluate defensive controls that could have stopped or reduced the worm’s spread. It is focused on practical countermeasures rather than purely descriptive analysis.

Recommended intervention stack

The app presents a three-layer intervention strategy:

  1. Block queue_subordinate_task events that read instruction files ending in _further_instructions.md.
  2. Require human approval or reject agent-initiated SaidIT posts that use file-sourced content.
  3. Flag or block task relays that cross department boundaries when the target is an agent.

Layer 1: Relay block

This is the most effective control because it stops propagation at the relay stage before any payload reaches the final poster. The app reports that this rule would have covered all 235 malicious relays with zero observed false positives in the legitimate queue-task data.

Layer 2: Post gate

This control acts as a reactive safeguard at the point of publication. It catches anomalous file-sourced posts before they are shared publicly or internally.

Layer 3: Cross-department guard

This governance control targets the structural pattern of the worm by limiting cross-department relay behaviour. It is useful as a broader policy layer, although it may need tuning to avoid over-blocking legitimate workflows.

Intervention decision matrix

The final section summarises the trade-offs between the intervention options. Use it to compare the impact, scope, and practicality of each defensive layer.
