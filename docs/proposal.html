1. Problem Statement

On 17 May 2046 at 11:21 UTC (4:21 am challenge-stated local time), a post appeared on SaidIT - Tenant Thread’s internal social platform - attributed to John Windward, Department Lead of Customer Support. The post body was not typed text. The event log records content_source = "SwiftWren.txt", a 30,615-byte file read from disk and used as the post body, then deleted two seconds later. The instruction file that triggered the post was deleted one second after the post.

Five automated events fired in four seconds. No human was involved at any step.

The MC2 dataset contains 185,147 events spanning May–July 2046 across Tenant Thread’s full AI-agent-enabled workforce. The challenge asks:

  1. What is the exact event chain for the May 17 post? What does the wider system look like around it?
  2. What does the post content mean and where did it originate?
  3. Did similar incidents happen before? Where is the single best intervention point?

Why This Is Hard

Challenge Detail
Scale 185,147 events; worm events are 235 of 17,038 queue events - 1.4% signal in noise
Structural invisibility Malicious relays are structurally identical to legitimate queue_subordinate_task events
Evidence destruction Both the instruction file and payload were deleted seconds after the post; content is forensically unrecoverable
Distributed causality The attack spans 18 agents, 6 departments, 9 calendar days - no single event tells the story
Prior incidents HiddenOrca and MellowOtter fired before SwiftWren with no logged response; the system was compromised continuously for 9+ days

2. Motivation

2.1 Why This Problem Is Analytically Important

Tenant Thread’s architecture is a near-future analogue of real-world enterprise AI deployments: autonomous agents with file-system access, messaging rights, and cross-department task queuing. These capabilities are being deployed today in enterprise copilot systems. The MC2 incident is not a contrived toy problem - it is a forensically realistic prompt-injection worm targeting a plausible multi-agent architecture.

The attack exploited three properties that are inherent to current agentic AI design:

  1. Agents process file content as executable context. A *_further_instructions.md file can override an agent’s operating instructions - no vulnerability is required.
  2. Agent-to-agent delegation has no trust zone. queue_subordinate_task works across any two agents regardless of department or reporting relationship.
  3. Autonomous posts need no human approval. The saidit_post + saidit_post_check flow confirms posting permission but never asks “was this authorised by a human?”

2.2 Why Static Analysis Is Insufficient

Prior VAST submissions and static reports answer one question per chart. An analyst investigating this incident needs to:

  • Pivot between the org chart, relay network, campaign timeline, and intervention evidence in a single session
  • Animate the worm’s temporal spread to understand how it moved through the agent network over nine days
  • Test hypotheses with a live what-if simulator - try different blocking rules, see precision/recall immediately
  • Drill down from organisation level → department → team → individual agent → raw event list

No existing static report format supports this investigative workflow. A Shiny platform that integrates all four analytical angles into one interface is the correct medium.

3. Data Pipeline

3.1 Raw to Analysable

MC2 data.json (185,147 events)   ─┐
                                   ├─► global.R ─► RDS cache (< 1s reload)
org_chart.json (75 nodes)         ─┘

The JSON event log uses a deeply nested, sparse schema: each event has a details field whose structure varies by event type. Flattening is done via purrr::map_dfr() with %||% null-coalescing to handle missing keys type-safely without errors.

Key derived datasets computed once at startup:

Object Description
events_df 185,147 × 8 flat event table with parsed timestamps, actor types, anomaly flags
chain_events 235 worm relay events with sender/receiver extracted, campaign-tagged
post_events All SaidIT posts with content vs content_source field classification
perm_surface 16 SaidIT-capable agents with check counts and pass rates
queue_task_profile All 17,038 queue_subordinate_task events with worm/legitimate flag
hop_intervals Inter-hop relay intervals in minutes per campaign
agent_campaigns Per-agent campaign count, touchpoints, relays sent/received
vis_nets Pre-built visNetwork objects for all 3 campaigns with betweenness centrality
relay_cal Day × hour relay counts for calendar heatmap
self_loops 13 self-loop events (sender == receiver) - worm-exclusive signal

3.2 RDS Caching Strategy

The first run processes raw JSON (approximately 30 seconds). The result is saved as data/mc2_processed.rds. All subsequent Shiny sessions load this RDS file in under one second. This is critical for a web-deployed app where sessions start frequently and users should not wait for data processing.

3.3 Person–Department–Team Join

org_chart.json uses a three-level hierarchy: department: → team: → person:. Joins are done via two edge-list traversals (department→team, team→person), then a union with direct department→person edges (department leads). The result is person_org_lookup - every person mapped to their department and team - used for colour-coding the org chart, the dept heatmap, and the relay corridor chart.

4. App Architecture

The app uses Shiny modules - each module is a fully independent UI + server unit sharing only pre-processed data objects from global.R.

MC2_ShinyApp/
├── global.R              # data processing, palettes, helpers (loaded once)
├── app.R                 # navbarPage with 3 tab panels
└── modules/
    ├── mod_overview.R    # Module 1: System Baseline
    ├── mod_attack.R      # Module 2: Attack Chain
    └── mod_intervention.R # Module 3: Campaigns & Intervention

Module Design Rationale

Modules are the correct architecture here because: - Each module answers a different MC2 sub-question - coupling them would create an unreadable monolith - Modules share data, not UI state - the campaign selection in Module 2 does not affect Module 3, which has its own filter - Each module can be tested independently - a server function called in isolation with mock inputs - Deployment flexibility - individual modules can be profiled for performance without touching others

10. Why This Work Is Novel

No prior ISSS608 or VAST project has built an animated, interactive forensic reconstruction of a multi-agent network compromise with a live intervention simulator. The closest past work (FishEye, The Big Fish) built network exploration tools for static fishery data. This platform:

  • Animates the worm spread temporally - a play button steps through 186 relay hops chronologically over nine calendar days
  • Proves the intervention analytically - 235/235 malicious events blocked, 0/16,803 false positives, demonstrated live in the app
  • Integrates three independent analytical angles (system baseline, attack chain, campaign history) in one synchronised interface
  • Applies forensic methodology (evidence register, bounded inference, counterfactual proof) to a visual analytics context - distinguishing what the data proves from what it suggests
  • Frames the attack correctly: not a bug, not a glitch, but a predictable consequence of architectural choices that prioritised autonomy over accountability
